> Baited Brothers(낚시당한 녀석들)의 현재 오프라인 지원 범위와 목표 정책

## 1. 결론

현재 앱은 오프라인 우선 앱이 아니다. 로그인 상태, 정적 도감, 일부 해양 정보 캐시와 로컬 알림은 오프라인에서도 사용할 수 있지만, 핵심 기능인 **AI 판별·조과 사진 업로드·조과 저장은 네트워크 연결이 필요하다.**

특히 조과 촬영 결과를 영구 보관하는 업로드 대기열이 없으므로, 낚시 현장에서 통신이 끊기면 촬영한 사진과 위치 정보가 유실될 위험이 있다.

[`field-catch-verification-v1.1.md`](./field-catch-verification-v1.1.md)의 오프라인 촬영 흐름은 목표 정책이며 현재 배포 버전에 구현된 동작이 아니다.

## 2. 현재 지원 범위

### 2.1 오프라인에서 가능한 기능

| 기능 | 현재 동작 | 제한 |
|------|-----------|------|
| 로그인 상태 유지 | Supabase 세션을 기기에 저장 | 토큰 갱신·재로그인은 네트워크 필요 |
| 한국 연안 60종 도감 | 앱에 포함된 폴백 데이터 사용 | 서버의 최신 수정사항은 반영되지 않음 |
| GPS 위치 획득 | 기기 위치 서비스 사용 | 환경에 따라 측위가 느리거나 실패할 수 있음 |
| 해안·내륙 판정 | 앱에 포함된 대한민국 해안 경계로 계산 | 관측소·주소·날씨 조회는 별도 네트워크 필요 |
| 기존 출조 알림 | 기기에 예약된 로컬 알림 실행 | 서버에서 변경된 일정은 동기화되지 않음 |
| 이전 해양 정보 | AsyncStorage의 최근 조회 결과 표시 | 최신 정보가 아니며 오래된 자료로 표시해야 함 |
| 일부 이미지 | OS 이미지 캐시에 남은 파일이 보일 수 있음 | 명시적인 오프라인 보장을 하지 않음 |
| 지도 화면 | 이미 캐시된 Apple 지도 영역이 보일 수 있음 | 지도 타일 다운로드·표시를 보장하지 않음 |

### 2.2 오프라인에서 불가능하거나 보장되지 않는 기능

- AI 어종 후보 판별
- 조과 원본·썸네일 업로드
- 조과 DB 저장
- 출조 생성·수정·완료·삭제
- 즐겨찾기 추가·삭제
- 최신 조과·출조·즐겨찾기·지도 핀 조회
- 새로운 물때·날씨·바람·파고·수온 조회
- 프로필이나 출조 커버 사진 변경
- 계정 로그인·가입·삭제

## 3. 현재 조과 촬영 흐름의 문제

현재 촬영 흐름은 다음과 같다.

```text
앱 카메라 촬영 + GPS 확보
  → 사진을 화면 상태에 보관
  → Supabase Edge Function으로 AI 판별
  → 사용자 어종 확정
  → 기존 인증 조과 조회
  → Supabase Storage에 원본·썸네일 업로드
  → user_catches 레코드 생성
```

네트워크가 끊기면 다음 문제가 발생한다.

1. AI 판별 요청이 실패한다.
2. 사용자는 정적 도감에서 직접 어종을 선택할 수 있지만 최종 저장은 실패한다.
3. 사진과 촬영 메타데이터는 영구 업로드 큐가 아닌 현재 화면 상태에만 존재한다.
4. 화면을 나가거나 앱이 종료되면 사진·좌표·촬영 시각을 복원할 수 있다고 보장할 수 없다.
5. 연결이 돌아와도 자동 재시도하지 않는다.
6. 사용자가 동일한 촬영 과정을 다시 수행해야 할 수 있다.

현재 `client_request_id`가 중복 조과 생성을 막는 데 사용되고 있으므로 향후 오프라인 재시도의 멱등성 키로 재사용할 수 있다.

## 4. 현재 데이터 조회의 문제

조과와 출조 목록은 앱 실행 시 Supabase에서 다시 조회한다. 네트워크 오류 시 영구 로컬 스냅샷으로 대체하지 않는다.

- 조과 조회가 실패하면 현재 조과 배열을 비운다.
- 출조 조회가 실패하면 현재 출조 배열을 비운다.
- 출조 일지 조회가 실패하면 일지와 집계를 비운다.
- 즐겨찾기는 서버 조회를 기본 데이터 소스로 사용한다.
- 서명된 이미지 URL 문자열은 캐시하지만 실제 이미지 파일의 오프라인 보관은 보장하지 않는다.

이 때문에 사용자가 출조 전에 확인했던 일정이나 기록도 현장에서 앱을 다시 열었을 때 보이지 않을 수 있다.

## 5. 현재 해양 정보 캐시

물때·날씨·바람·파고·수온 정보는 좌표를 소수 둘째 자리로 반올림한 키로 AsyncStorage에 저장한다.

- 최근 30분 이내 데이터는 캐시에서 즉시 반환한다.
- 30분이 지난 데이터는 서버 새로고침을 시도한다.
- 서버 호출이 실패해도 기존 데이터가 있으면 화면에 유지한다.
- 오래된 데이터에는 `stale` 상태를 표시할 수 있다.
- 한 번도 조회하지 않은 장소는 오프라인에서 정보를 제공할 수 없다.

출조 안전과 관련된 정보이므로 오래된 자료를 최신 관측값처럼 표시해서는 안 된다. 관측·예보 시각과 `마지막 업데이트`를 항상 함께 보여주는 것이 원칙이다.

## 6. 목표 오프라인 정책

### 6.1 핵심 원칙

1. 촬영은 네트워크 상태와 관계없이 완료할 수 있어야 한다.
2. 사진·GPS·촬영 시각은 셔터 직후 영구 로컬 저장소에 보관한다.
3. 현장 인증은 업로드 위치가 아니라 촬영 당시 정보를 기준으로 한다.
4. 온라인 복구 후 AI 판별과 서버 저장을 이어간다.
5. 앱 종료·재실행·기기 재부팅 후에도 대기 작업을 복구한다.
6. 모든 재시도는 중복 조과를 만들지 않아야 한다.
7. 사용자가 현재 상태와 필요한 다음 행동을 알 수 있어야 한다.

### 6.2 목표 조과 흐름

```text
현장에서 앱 카메라 촬영
  → 사진을 앱 영구 저장소에 복사
  → GPS·정확도·촬영 시각·출조 ID 저장
  → 로컬 작업을 pending_analysis로 생성
  → 네트워크 복구 감지
  → AI 후보 판별
  → 사용자가 어종 확정
  → 사진 업로드
  → 서버 조과 생성
  → 현장 인증 검증
  → 로컬 원본 및 완료 작업 정리
```

AI 후보 선택 전에 다시 오프라인이 되더라도 작업을 삭제하지 않는다. 사용자가 후보를 확정한 뒤 업로드가 실패한 경우에는 `pending_upload` 상태로 유지한다.

## 7. 오프라인 작업 상태

| 상태 | 의미 | 사용자 행동 |
|------|------|-------------|
| `capturing` | 사진·위치 확보 중 | 잠시 대기 |
| `pending_analysis` | 촬영 보존 완료, AI 판별 대기 | 연결 후 자동 판별 또는 수동 재시도 |
| `needs_confirmation` | AI 후보 준비 완료 | 어종 선택 필요 |
| `pending_upload` | 어종 확정 완료, 서버 저장 대기 | 연결 후 자동 업로드 |
| `uploading` | 사진과 레코드 업로드 중 | 앱을 닫아도 작업 유지 |
| `verification_pending` | 서버 현장 인증 판정 중 | 결과 대기 |
| `completed` | 저장과 인증 처리 완료 | 기록 열기 |
| `failed_retryable` | 일시적인 네트워크·서버 오류 | 다시 시도 |
| `failed_permanent` | 손상된 사진·인증 불가 등 | 일반 기록 전환 또는 삭제 |

## 8. 로컬 저장 항목

각 오프라인 작업은 최소한 다음 정보를 보관한다.

| 항목 | 설명 |
|------|------|
| `local_id` | 로컬 작업 식별자 |
| `client_request_id` | 서버 중복 저장 방지 키 |
| `user_id` | 촬영한 사용자 |
| `trip_id` | 연결된 출조, 선택 |
| `local_image_path` | 앱 영구 저장소의 원본 |
| `local_thumbnail_path` | 로컬 목록용 썸네일 |
| `captured_at` | 셔터 시각 |
| `latitude` / `longitude` | 셔터 순간 좌표 |
| `location_accuracy_m` | 수평 위치 정확도 |
| `location_captured_at` | 위치 측정 시각 |
| `capture_method` | `live_camera` 등 촬영 경로 |
| `selected_fish_id` | 사용자가 확정한 어종, 선택 전에는 null |
| `candidate_fish_ids` | AI 후보, 판별 전에는 빈 배열 |
| `size_cm` / `memo` | 사용자 입력 |
| `status` | 오프라인 작업 상태 |
| `attempt_count` | 업로드 시도 횟수 |
| `last_error` | 마지막 실패 사유 |
| `created_at` / `updated_at` | 작업 생성·변경 시각 |

사진 파일은 OS가 임의로 정리할 수 있는 카메라 임시 경로나 캐시 디렉터리에만 두지 않는다.

## 9. 동기화 원칙

- 앱 실행, 포그라운드 복귀와 네트워크 복구 시 대기열을 확인한다.
- 동일한 `client_request_id`에 대한 서버 레코드가 있으면 성공으로 간주한다.
- 원본과 썸네일 업로드 경로는 작업 ID에서 결정적으로 생성한다.
- 부분 업로드 후 실패해도 다음 재시도에서 중복 파일을 만들지 않는다.
- 지수 백오프를 적용하고 사용자의 `지금 다시 시도`를 제공한다.
- 로그아웃 시 다른 계정의 대기 작업을 업로드하지 않는다.
- 계정 전환 시 작업 소유자를 확인한다.
- 성공 확인 전에는 로컬 원본을 삭제하지 않는다.
- 영구 실패 작업은 자동 삭제하지 않고 사용자가 일반 기록 전환 또는 삭제를 선택하도록 한다.

Supabase Storage와 DB 저장은 하나의 원자적 트랜잭션이 아니므로 부분 성공을 전제로 복구 흐름을 설계해야 한다.

## 10. 출조 전 오프라인 준비

예정 출조가 있는 사용자를 위해 출조 전에 필요한 정보를 기기에 준비한다.

- 예정 출조와 장소 정보
- 즐겨찾기 장소
- 한국 연안 60종 도감과 로컬 일러스트
- 해당 장소의 최신 해양 정보와 관측 시각
- 최근 조과 및 출조 요약
- 로컬 출조 알림

앱은 출조 상세에서 `오프라인 준비 완료` 여부와 마지막 동기화 시각을 보여줄 수 있다.

지도 타일은 Apple 지도 캐시에 의존하므로 완전한 오프라인 지도를 약속하지 않는다. 대신 출조 장소명, 좌표, 미니 방향 정보와 저장된 해양 정보는 지도 타일이 없어도 확인할 수 있어야 한다.

촬영 직후:

> 현장 촬영을 안전하게 저장했어요
> 연결되면 AI 판별과 업로드를 이어갑니다.

AI 판별 대기:

> 인터넷 연결을 기다리고 있어요
> 사진과 촬영 위치는 이 기기에 보관되어 있습니다.

어종 확인 필요:

> AI 후보가 준비됐어요
> 어종을 확인하면 업로드를 완료할 수 있습니다.

업로드 실패:

> 아직 업로드하지 못했어요
> 사진은 안전하게 보관 중이며 연결되면 다시 시도합니다.

오래된 해양 정보:

> 현재 오프라인 상태로 최신 관측 정보가 아닐 수 있습니다.

## 12. 구현 우선순위

1. 촬영 사진을 앱 영구 저장소에 복사

4. 마지막 동기화 시각 표시

1. 부분 업로드 복구와 고아 Storage 객체 정리

3. 계정 전환·로그아웃 시 대기열 보호

5. 동기화 실패 분석 이벤트

## 13. 출시 전 인수 조건

- 완전한 비행기 모드에서 사진과 GPS를 촬영해 앱 영구 저장소에 보관할 수 있다.
- 촬영 직후 앱을 강제 종료해도 다음 실행에서 작업을 복구한다.
- 네트워크가 복구되면 AI 판별을 이어간다.
- 어종 확정 후 업로드가 실패해도 사진과 입력값을 잃지 않는다.
- 같은 작업을 여러 번 재시도해도 조과가 한 건만 생성된다.
- 낚시터에서 촬영하고 내륙에서 업로드해도 촬영 당시 위치로 현장 인증한다.
- 로그아웃하거나 계정을 바꿔도 다른 사용자의 작업을 잘못 업로드하지 않는다.
- 조과·출조 조회 실패 시 마지막 로컬 데이터를 빈 화면으로 대체하지 않는다.
- 오래된 날씨·수온·물때에는 마지막 갱신 시각과 오프라인 상태를 표시한다.
- 저장 완료 전까지 로컬 원본을 삭제하지 않는다.

## 14. 최종 제품 원칙

> **통신이 끊겨도 현장에서 얻은 사진과 촬영 증거는 절대 잃지 않는다.**

- 촬영과 증거 보존은 오프라인에서 끝나야 한다.
- AI 판별과 서버 업로드는 연결 후 이어갈 수 있어야 한다.
- 업로드 장소는 현장 인증 여부에 영향을 주지 않는다.
- 최신성이 중요한 해양 정보는 캐시 여부와 관측 시각을 명확히 표시한다.
- 사용자는 자신의 기록이 안전하게 저장됐는지 항상 알 수 있어야 한다.
